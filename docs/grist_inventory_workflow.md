# Grist Inventory Workflow

This project uses Grist as the operational UI for inventory tracking while keeping durable snapshots in the repo. Day-to-day operations (syncing, computing the shopping list, finding products) run via Claude Code skills. This document covers the one-time setup and the data model.

## Ownership rules

- `bin-store-model/` plus the extraction mapping in `scripts/grist_inventory/extraction.py` are the source of truth for the generated `cut_list`.
- `data/cut_list.json` is generated. Do not edit it manually.
- `data/shopping_list.json` is generated by `scripts/compute_shopping_list.py`. Product links are filled in by the `bin-store-shopping` skill.
- The Grist `cut_list` table is a generated mirror of repo state, except for the editable `completed` checkbox.
- The Grist `inventory` table is where manual stock edits happen.
- `data/inventory.json` is the repo snapshot of the live inventory data.

## Table sync directions

| Grist table          | Direction     | Authored in                                        |
| -------------------- | ------------- | -------------------------------------------------- |
| `cut_list`           | local → Grist | Repo (OpenSCAD extraction)                         |
| `cut_list.completed` | Grist → local | Grist (pulled, merged, and rewritten on each push) |
| `shopping_list`      | local → Grist | Repo (`compute_shopping_list.py` + shopping skill) |
| `inventory`          | Grist → local | Grist (manual stock entry)                         |
| `material_rules`     | Grist → local | Grist (optional purchasing hints)                  |

The sync script (`scripts/sync_grist_tables.py`) enforces these directions via its `TABLE_SPEC` registry. Running `python3 scripts/sync_grist_tables.py` with no flags syncs every registered table in its default direction.

## Initial setup (one-time)

### 1. Create the Grist document

1. Sign in at [https://docs.getgrist.com](https://docs.getgrist.com) on the free plan.
2. Create a new empty document.
3. Copy the doc ID from the URL — the segment after `/doc/`.
4. From the profile menu → Profile Settings, generate an API key and copy it.

### 2. Set environment variables

Copy `.env.sample` to `.env` at the repo root and fill in the values:

```bash
cp .env.sample .env
```

Then edit `.env`:

```
GRIST_API_KEY=...
GRIST_DOC_ID=...
# Optional, only for self-hosted Grist:
# GRIST_BASE_URL=https://<your-grist-host>/api
```

`scripts/sync_grist_tables.py` loads `.env` automatically via `python-dotenv`. `.env` is gitignored; `.env.sample` is committed as a template. Real shell exports (`export GRIST_API_KEY=...`) still work and take precedence over `.env`.

### 3. Bootstrap tables and columns

From the repo root:

```bash
python3 scripts/sync_grist_tables.py --bootstrap-schema
```

Reads `data/grist_schema.json` and creates the `projects`, `cut_list`, `inventory`, `shopping_list`, and `material_rules` tables with their columns. Idempotent for missing tables/columns.

### 4. Annotate the cut_list table

In the Grist UI, open the `cut_list` table and add a short description:

> Generated from OpenSCAD / code. Only the `completed` checkbox is editable here — every other field is overwritten on the next sync.

### 5. Initial sync

Seed the doc with the current repo state:

```bash
python3 scripts/sync_grist_tables.py
```

Confirm the `cut_list` rows appear in Grist. Inventory/material_rules will pull empty until you add rows in the Grist UI.

## Required environment variables

- `GRIST_API_KEY`: API key from the Grist profile settings page.
- `GRIST_DOC_ID`: document id from the doc URL.
- `GRIST_BASE_URL`: optional; defaults to `https://docs.getgrist.com/api`.

## Hosted vs self-hosted Grist

The default config targets hosted Grist on the free plan. Free-plan limits cover this project's size, API-key auth is supported, and it avoids running another local service. To run against a self-hosted instance later, change `GRIST_BASE_URL`.

## Daily usage

Day-to-day operations run through Claude Code skills in `.claude/skills/`:

- `**bin-store-sync**` — push `cut_list`/`shopping_list`, pull `inventory`/`material_rules`. Trigger with "sync Grist", "refresh the cut list", or "pull inventory".
- `**bin-store-shopping**` — auto-sync, compute outstanding cuts (cut list minus inventory), search B&Q for each outstanding section, write product links to `shopping_list`, and push back to Grist. Trigger with "what do I need to buy" or "refresh the bin store shopping list".

Invoking the skill is the recommended entrypoint; the underlying scripts remain available for manual runs.

## Repo files

- `scripts/extract_cut_list.py`: regenerates `data/cut_list.json` from the OpenSCAD model.
- `scripts/sync_grist_tables.py`: bootstraps and syncs Grist tables per the direction registry.
- `scripts/compute_shopping_list.py`: computes outstanding cuts (greedy per-cut fit with 3 mm kerf and 150 mm minimum usable offcut) and seeds `data/shopping_list.json`.
- `scripts/export_reports.py`: exports CSV/Markdown reports to `data/exports/`.
- `scripts/grist_inventory/`: shared package — `common` (paths, snapshot I/O), `extraction` (OpenSCAD parser), `grist_api` (REST client), `requirements` (shortfall algorithm), `schema`.
- `data/grist_schema.json`: canonical schema for the Grist base.
- `data/cut_list.json`, `data/inventory.json`, `data/shopping_list.json`, `data/material_rules.json`: snapshots.

## Current extraction scope

The generated `cut_list` covers:

- frame timber
- brace lengths derived from modeled geometry
- roof battens and roof deck
- shelf battens plus inferred shelf panels
- modeled hardboard and featheredge cladding
- door frame timber and door featheredge panel

Hardware and fixings remain out of scope.
